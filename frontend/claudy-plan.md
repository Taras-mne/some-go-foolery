## Фаза 1 — Relay-сервер (1-2 недели)

Это сердце проекта — начинаем отсюда, потому что демон ни к чему не подключится без relay.

### 1.1 База (3-4 дня)

- [ ] Инициализировать Go модуль в `relay/`
- [ ] База данных: SQLite через `gorm` (достаточно для MVP)
  - Таблица `users`: id, email, password_hash, plan, created_at
  - Таблица `devices`: id, user_id, name, fingerprint, last_seen, created_at
- [ ] Эндпоинты авторизации:
  - `POST /v1/auth/register` — email + password → создать юзера
  - `POST /v1/auth/login` → JWT токен (срок 30 дней)
- [ ] Middleware: проверка JWT на защищённых роутах
- [ ] Проверить через curl: регистрация и логин работают

### 1.2 Device management (2 дня)

- [ ] При каждом запросе с токеном — регистрировать device fingerprint
  - Fingerprint = SHA256(User-Agent + уникальный device_token из заголовка)
  - Device_token генерируется демоном при первом запуске, хранится локально
- [ ] Проверка лимита устройств по плану:
  - Free → 2 устройства
  - Personal → 5
  - Family → 15
- [ ] При превышении → `403 DEVICE_LIMIT_REACHED`
- [ ] Эндпоинты управления устройствами:
  - `GET /v1/devices` — список устройств юзера
  - `DELETE /v1/devices/{id}` — удалить устройство
  - `PATCH /v1/devices/{id}` — переименовать

### 1.3 WebSocket туннель (3-4 дня)

Это самая сложная часть фазы 1.

- [ ] Демон коннектится к relay по WebSocket (`WS /v1/tunnel` с JWT в заголовке)
- [ ] Relay хранит map: `user_id → websocket connection`
- [ ] Входящий HTTP запрос на `dav.claudy.app/{user_id}/...`:
  1. Relay находит туннель по `user_id`
  2. Пересылает HTTP запрос через WebSocket как бинарное сообщение
  3. Демон обрабатывает запрос локально, отдаёт ответ обратно через WebSocket
  4. Relay возвращает ответ клиенту
- [ ] Обработка edge cases:
  - Демон оффлайн → `503 Source offline`
  - Таймаут ответа → `504 Gateway timeout`
  - Переподключение демона при обрыве

### 1.4 Деплой relay (1 день)

- [ ] `Dockerfile` для relay
- [ ] `deploy/docker-compose.yml`:
  ```yaml
  services:
    relay:
      build: ../relay
      environment:
        - JWT_SECRET=${JWT_SECRET}
        - DB_PATH=/data/claudy.db
      volumes:
        - ./data:/data
    nginx:
      image: nginx
      # конфиг с SSL терминацией
    certbot:
      image: certbot/certbot
  ```
- [ ] Nginx конфиг: `dav.claudy.app` → relay, SSL через Let's Encrypt
- [ ] Задеплоить на VPS, проверить HTTPS

## Фаза 2 — Go-демон (1-2 недели)

### 2.1 Ядро (3-4 дня)

- [ ] Инициализировать Go модуль в `daemon/`
- [ ] Подключить `hacdias/webdav` как зависимость
- [ ] Локальный WebDAV сервер на выбранной папке (read/write)
- [ ] Конфиг файл (`~/.claudy/config.json`):
  ```json
  {
    "folder": "/Users/user/CloudFolder",
    "token": "jwt...",
    "device_token": "uuid..."
  }
  ```
- [ ] Логин: `POST /v1/auth/login` → сохранить JWT в конфиг
- [ ] Проверить: WebDAV сервер отвечает локально на `localhost:6080`

### 2.2 Туннельный клиент (2-3 дня)

- [ ] WebSocket клиент: коннект к `wss://relay.claudy.app/v1/tunnel`
- [ ] Слушать входящие HTTP-запросы через WebSocket
- [ ] Проксировать их в локальный WebDAV сервер
- [ ] Отправлять ответы обратно через WebSocket
- [ ] Автоматическое переподключение с exponential backoff при обрыве
- [ ] Проверить: файлы доступны с другого устройства через relay

### 2.3 Автозапуск (1-2 дня)

- [ ] **Windows:** запись в реестр `HKCU\Software\Microsoft\Windows\CurrentVersion\Run`
- [ ] **macOS:** создать `~/Library/LaunchAgents/app.claudy.daemon.plist`
- [ ] **Linux:** создать `~/.config/systemd/user/claudy.service`
- [ ] Команды `claudy start` / `claudy stop` / `claudy status`

### 2.4 Tray-иконка (1-2 дня)

- [ ] Подключить `github.com/getlantern/systray`
- [ ] Иконка в трее с индикатором статуса (зелёный/красный)
- [ ] Меню: статус подключения, открыть папку, открыть дашборд, выйти
- [ ] При запуске без конфига — открыть браузер на страницу входа

### 2.5 Инсталляторы (2-3 дня)

- [ ] GitHub Actions workflow для сборки:
  ```yaml

  jobs:
    build:
      matrix:
        os: [windows-latest, macos-latest, ubuntu-latest]
  ```
- [ ] **Windows:** WiX Toolset → `.msi`
- [ ] **macOS:** `create-dmg` → `.dmg` с иконкой

## Критический путь

Если что-то пойдёт не так — здесь будут проблемы:

1. **WebSocket туннель (Фаза 1.3)** — самая сложная техническая часть. Если застрянешь — попроси агента написать сначала упрощённую версию без multiplexing, потом улучшить.

2. **Автозапуск на macOS (Фаза 2.3)** — Apple периодически меняет правила для LaunchAgent. Протестировать на реальном Mac обязательно.

3. **Инсталлятор Windows (Фаза 2.5)** — подписывание кода ($70-200/год) нужно чтобы Windows не показывал страшное предупреждение. Для MVP можно без него, но пользователи будут пугаться.
