# Project Brief — Claudy

## Суть

Приложение, которое превращает любой старый компьютер в личное облако.
Пользователь скачивает инсталлятор под свою ОС, выбирает папку — и папка становится доступна со всех его устройств как сетевой диск (через WebDAV). Никаких VPS, никакого проброса портов, никаких технических знаний. Работает на Windows, macOS и Linux.

## Проблема

Nextcloud и аналоги требуют сервера, навыков настройки, командной строки.
Google Drive / iCloud — чужие серверы, платная подписка, нет приватности.
Syncthing / Tailscale — нет единого аккаунта, сложная первичная настройка.

## Решение

```
[Старый ноут / ПК]               [Relay-сервер]        [Устройства пользователя]
  daemon + WebDAV-сервер  ←→   reverse tunnel   ←→   монтируют как сетевой диск
  выбранная папка                авторизация            iOS / Android / Win / Mac / Linux
```

Ключевые принципы:
- **Кросс-платформенный инсталлятор** — Windows (.msi), macOS (.dmg), Linux (.deb / .rpm / AppImage)
- **Один демон** — установил, выбрал папку, готово; работает в фоне
- **WebDAV** — нативная поддержка во всех ОС без дополнительных приложений
- **Аккаунт** — единая точка входа для всех устройств пользователя
- **Relay без VPS у пользователя** — NAT traversal через наш сервер

## Технический стек

| Компонент | Технология | Обоснование |
|---|---|---|
| Демон (клиент-сервер) | Go | Кросс-компиляция под все ОС, один бинарник |
| Инсталлятор | .msi / .dmg / .deb+AppImage | Нативный UX на каждой платформе |
| WebDAV ядро | `hacdias/webdav` (MIT) | Готовое, надёжное, open source |
| NAT traversal v1 | Reverse WebSocket tunnel | Простота, работает везде |
| NAT traversal v2 | Tailscale SDK или WireGuard + relay fallback | P2P когда возможно, relay когда CGNAT |
| Relay сервер | Go / Node.js на VPS | Проксирует WebDAV трафик |
| Авторизация | JWT + аккаунт на нашем сервере | Единый вход со всех устройств |
| Транспорт | TLS везде | Обязательно |

## Инфраструктура и затраты

**Что нужно для деплоя:**
- VPS с публичным IP — Hetzner / DigitalOcean / Timeweb, от $5-6/мес
- Домен — ~$10/год (например `claudy.app`)
- Docker Compose + Let's Encrypt — разворачивается одной командой

**Трафик:**

| Пользователей | Трафик/мес | Стоимость (Hetzner) |
|---|---|---|
| 100 | ~500 ГБ | ~$5 |
| 1 000 | ~5 ТБ | ~$50 |
| 10 000 | ~50 ТБ | ~$500 |

Hetzner включает 20 ТБ/мес на минимальном сервере бесплатно.
При 1000 платных юзеров по $3/мес → $3000 выручки vs $50-100 расходов на трафик.

**Рычаги для контроля трафика:**
- Лимит устройств на free плане (уже есть)
- Троттлинг скорости на free tier
- P2P в v2 снизит нагрузку на relay в разы



- [ ] Go-демон: поднимает WebDAV, коннектится к relay
- [ ] Инсталляторы: Windows (.msi), macOS (.dmg), Linux (.deb / AppImage)
- [ ] relay: проксирует запросы, проверяет токен
- [ ] веб-страница: регистрация / логин, инструкция подключения
- [ ] инструкция подключения для iOS / Android / Windows / macOS / Linux

## Монетизация

Лимит по количеству устройств-получателей (тех, что монтируют папку).
Источник (ноут с расшаренной папкой) — один на всех планах.

| План | Устройств-получателей | Цена |
|---|---|---|
| Free | 2 | $0 |
| Personal | 5 | $3/мес |
| Family | 15 | $7/мес |

**Защита от обхода через перелогин — модель Spotify:**
- При первом подключении устройство регистрируется в аккаунте по device fingerprint
- В дашборде пользователь видит список "Мои устройства" и управляет ими вручную
- Новое устройство сверх лимита → предложение апгрейда, не молчаливый отказ

## Не входит в MVP

- Нативные мобильные приложения (нужны в v2)
- E2E шифрование
- Несколько источников одновременно
- Биллинг / Stripe
- Общий доступ к диску/файлу (в v3)

## Roadmap

### v1 — MVP
Доступ к папке на ноуте с любого устройства через WebDAV. Аккаунт, relay, инсталлятор, лимиты по устройствам.

**Веб-интерфейс (браузерный доступ):**
Полноценный файловый менеджер в браузере — как Google Drive или Dropbox. Пользователь заходит на `claudy.app`, логинится, видит свои файлы. Без установки приложения, с любого устройства.

Минимальный набор для v1:
- Список файлов и папок с иконками, размером, датой изменения
- Навигация по папкам
- Скачать файл
- Загрузить файл (drag & drop)
- Создать папку
- Удалить файл / папку
- Переименовать

Технически: веб-интерфейс делает запросы к relay, который проксирует их в туннель к daemon — тот же механизм что и WebDAV, только через REST/JSON API вместо WebDAV-протокола. Либо relay умеет сам рендерить WebDAV-ответы в JSON для фронтенда.

Это критично для мобильного доступа (iOS/Android без нативного приложения) и для пользователей которые не хотят настраивать сетевой диск.

### v2 — Автовыгрузка фото + P2P

**Автовыгрузка фото:**
- Нативные приложения iOS и Android (WebDAV не умеет слушать камеру в фоне)
- Фоновый сервис на телефоне: следит за новыми фото и загружает на источник
- Веб-галерея или просмотрщик в дашборде

**Wake-on-LAN:**
Если ноут спит — приложение шлёт магический WoL пакет через relay, ноут просыпается, выгрузка начинается.

| Платформа | WoL |
|---|---|
| Windows | ✅ Надёжно (включить в BIOS + сетевой карте) |
| Linux | ✅ Надёжно (ethtool) |
| macOS ноут | ⚠️ Ненадёжно (Apple ограничивает с 2013) |
| Mac Mini / iMac | ✅ Надёжно |

В UI показывать статус "источник спит" + кнопку "разбудить" с предупреждением для Mac.

**P2P (WireGuard / Tailscale SDK):**
- Прямое соединение устройство ↔ ноут, минуя relay
- Relay остаётся как fallback для CGNAT случаев (~20-40% пользователей)
- Трафик через relay E2E зашифрован — мы не видим содержимое
- Маркетинговый аргумент: "Ваши файлы только ваши"
- Снижает нагрузку на relay в разы
- **Не libp2p** — тянет web3-ассоциации, WebDAV поверх него сложно склеить


### v3 — Галерея, шаринг и расширенные функции
- Веб-просмотрщик фото с поиском и альбомами
- E2E шифрование
- Несколько источников на одном аккаунте
- Биллинг / Stripe

**Общий доступ к диску / файлу:**

Два уровня шаринга:

**1. Шаринг всего диска** — пользователь создаёт дополнительный аккаунт (или приглашает другого пользователя Claudy), и тот подключает чужой диск как сетевой. Relay выдаёт токен с правами read-only или read-write на конкретный `user_id`. Полезно для семьи, небольшой команды, Obsidian-vault на двоих.

**2. Шаринг отдельной папки или файла** — пользователь указывает путь внутри своего диска и генерирует ссылку с токеном. Relay проверяет токен и ограничивает доступ только этим путём. Два режима:
- **Публичная ссылка** — открывается в браузере как веб-страница (скачать / просмотреть), без аккаунта
- **Приватная ссылка** — только для конкретного пользователя Claudy, монтируется как WebDAV

**Технически:** в relay добавляется таблица `shares` (share_token → owner_id, path, permissions, expires_at). При запросе relay проверяет токен, подставляет нужный туннель и ограничивает путь. Веб-просмотр файла — отдельный endpoint, рендерит HTML со списком файлов или превью.

**Монетизация:** шаринг только на платных тарифах. Бесплатный план — только личный доступ.

### v4 — Нативный десктопный клиент (FUSE / Dokan)

WebDAV через Finder/Explorer работает медленно не из-за протокола, а из-за устаревшего клиента Apple и Microsoft. Решение — собственный десктопный клиент, который монтирует виртуальный диск на уровне ОС, а под капотом использует любой оптимизированный протокол напрямую с daemon.

**Как работает:**
- **macOS / Linux** — FUSE (Filesystem in Userspace): клиент монтирует виртуальный диск через системный API, пользователь видит его в Finder как обычный диск, но I/O идёт через наш протокол
- **Windows** — Dokan (аналог FUSE для Windows) или WinFSP: то же самое, диск появляется в Проводнике как сетевой

**Почему это быстрее WebDAV:**
- Полный контроль над буферами и размером чанков
- Delta-синхронизация — передаём только изменённые блоки файла, не весь файл
- Нет лишних round-trip'ов (LOCK, PROPFIND перед каждой операцией)
- Можно использовать бинарный протокол вместо XML

**Именно так устроены** Dropbox, Google Drive, iCloud Drive — у каждого свой виртуальный диск поверх собственного протокола синхронизации.

**Сложность:** высокая. FUSE-драйверы требуют системных разрешений, сложны в отладке и требуют отдельного инсталлятора с правами администратора. Это v4, после того как продукт найдёт аудиторию и WebDAV-ограничения станут реальной болью пользователей.

### v5 — Совместная работа (WOPI + Office Online)

WebDAV поддерживает только последовательный доступ (один редактирует, остальные ждут). Для полноценной совместной работы нужен отдельный протокол.

**Выбранный путь: WOPI (Web Application Open Platform Interface)**

Протокол Microsoft, который используют SharePoint, OneDrive, Nextcloud. Сервер реализует несколько HTTP-эндпоинтов (WOPI API), а Office Online (бесплатный, хостится Microsoft) берёт на себя весь движок совместного редактирования — операционные преобразования, курсоры, конфликты. Пользователи открывают файл в браузере, видят курсоры друг друга как в Google Docs.

**Плюсы:**
- Полноценная совместная работа в Word, Excel, PowerPoint
- Всё в браузере — ничего не нужно устанавливать на устройства
- Microsoft уже решил все сложные задачи (OT, конфликты, курсоры)
- Именно так работает Nextcloud: "открыть в Office Online" из интерфейса

**Минусы:**
- Нужно зарегистрироваться в Microsoft Cloud Storage Partner Program
- Обязательны HTTPS и публичный домен (у нас есть)
- Файлы проходят через серверы Microsoft для рендеринга (не E2E)

**Альтернатива без Microsoft:** Collabora Online — LibreOffice в браузере, open source, тоже использует WOPI, можно поднять self-hosted. Потеря в качестве UI относительно Office Online, выигрыш в приватности.

# Agent Skills

## Agent 1 — Backend / Core

**Зона ответственности:** кросс-платформенный демон на Go, WebDAV-сервер, туннельный клиент, инсталляторы

**Контекст:**
- Язык: Go
- Основа: `golang.org/x/net/webdav` или `hacdias/webdav`
- Один Go-бинарник, кросс-компилируется под все платформы (`GOOS=windows/darwin/linux`)
- Инсталляторы: `.msi` (Windows, через WiX Toolset), `.dmg` (macOS), `.deb` + AppImage (Linux)
- При запуске: GUI для выбора папки, затем демон в фоне

**Ключевые задачи:**
1. WebDAV сервер поверх выбранной папки (read/write)
2. WebSocket клиент для коннекта к relay серверу
3. Авторизация: получение токена по логину/паролю, хранение локально
4. Автозапуск при старте системы:
   - Windows: реестр / Task Scheduler
   - macOS: LaunchAgent (plist)
   - Linux: systemd user service
5. Tray-иконка с индикатором статуса (библиотека `fyne` или `systray`)
6. CI/CD: GitHub Actions для сборки всех платформ и публикации релизов

**Интерфейсы с другими агентами:**
- REST API relay-сервера: `POST /auth/login`, `GET /tunnel/connect`
- Формат туннельного протокола (согласовать с DevOps агентом)

## Agent 2 — Relay Server / DevOps

**Зона ответственности:** relay-сервер, инфраструктура, деплой

**Контекст:**
- Язык: Go или Node.js
- Хостинг: любой VPS с публичным IP
- Должен держать тысячи одновременных WebSocket соединений

**Ключевые задачи:**
1. WebSocket сервер для приёма туннелей от демонов
2. HTTP/WebDAV reverse proxy: входящий запрос → нужный туннель по user_id
3. Аутентификация: выдача JWT токенов, проверка при каждом запросе
4. **Device management:**
   - При первом подключении устройства — регистрация по fingerprint (User-Agent + IP + случайный токен, хранится локально на устройстве)
   - Проверка лимита устройств по плану пользователя
   - При превышении лимита — ответ `403` с кодом `DEVICE_LIMIT_REACHED`
   - API для управления устройствами: просмотр, удаление, переименование
5. TLS терминация (Let's Encrypt / certbot)
6. Docker Compose конфиг для деплоя

**Безопасность:**
- Rate limiting на авторизацию
- Изоляция туннелей между пользователями (пользователь A не видит данные B)
- Логирование попыток несанкционированного доступа

**Интерфейсы:**
- `POST /auth/register` — регистрация
- `POST /auth/login` → JWT
- `WS /tunnel/{user_id}` — туннель от демона
- `ANY /dav/{user_id}/*` — WebDAV запросы от устройств

## Agent 3 — Frontend / Web

**Зона ответственности:** веб-сайт, страница входа, онбординг

**Контекст:**
- Стек: React или чистый HTML/CSS/JS (минимум зависимостей)
- Аудитория: обычные пользователи, не технари
- Tone: простой, дружелюбный, доверие к приватности

**Ключевые задачи:**
1. Лендинг с объяснением что это и зачем
2. Страница регистрации / логина
3. Дашборд:
   - Статус подключения источника (ноута)
   - Адрес WebDAV для подключения, кнопка "скопировать"
   - **Список устройств** — название, последняя активность, кнопка удалить (как у Spotify)
   - Индикатор плана и оставшихся слотов устройств ("2 из 2 устройств — Upgrade")
4. Страница онбординга с инструкциями подключения под каждую платформу:
   - iOS: Settings → Files → Connect to Server
   - Android: шаги для популярных файловых менеджеров
   - Windows: Map Network Drive
   - macOS: Finder → Connect to Server
   - Linux: Nautilus / Thunar → Connect to Server, или mount через davfs2

**UX принципы:**
- Один экран — одно действие
- Адрес WebDAV всегда на виду, кнопка "скопировать"
- Статус "онлайн/оффлайн" для ноута с которого расшарена папка

## Общие соглашения для всех агентов

- Язык кода: Go для бинарников, TypeScript для веба
- Все секреты через переменные окружения, не хардкод
- Версионирование API: `/v1/`
- Ошибки в формате `{"error": "message", "code": "ERROR_CODE"}`
- TLS обязателен на всех эндпоинтах в продакшене
